1. Architecture Overview
1.1 What Changed from v1
The v1 architecture hardcoded 24 machines across Series H1–H8. The referee knew exactlywhich ports to poll and which SSH targets to reach because everything was written into
.env at deploy time.
v2 inverts this. An organizer uploads any container they want. The platform stores it in a
registry, deploys it to a node, and the referee adapts its poll list by reading from the registry
at runtime instead of from a static config.

2. Machine Registry
2.1 What It Is
The registry is a SQLite database that is the single source of truth for every machine on the
platform. The referee reads from it. The deployer writes to it. The API exposes it.

3. Machine Definition Spec
3.1 koth-machine.yaml
Every machine on the platform, regardless of how it was uploaded, is described by this
schema. Organizers who use the web UI form submit data that is normalized into this format
before storage.
# koth-machine.yaml — complete reference
# ── Identity ──────────────────────────────────────────────────
name: "Vulnerable Nginx"
# Required. Shown in the participant board and admin dashboard.
# Max 64 characters. Must be unique within a competition.
description: "Nginx with a misconfigured sudo rule. Two-stage exploit."
# Optional. Shown to participants. Plain text only; no Markdown.
difficulty: medium

3.3 Normalization from compose_file
When an organizer uploads a docker-compose.yml, the platform reads the first service
entry and extracts:
# platform-server/engine/normalizer.py
import yaml  # always safe_load — see Section 12.1
def normalize_compose(compose_path: str) -> dict:
    with open(compose_path) as f:
        data = yaml.safe_load(f)
    services = data.get("services", {})
    if not services:
        raise ValueError("compose_file has no services defined")
    first_service = next(iter(services.values()))
    image = first_service.get("image")
    ports = []
    for p in first_service.get("ports", []):
        # docker-compose short syntax: "8080:80"
        if isinstance(p, str) and ":" in p:
            host, container = p.split(":", 1)
            ports.append({"host": int(host), "container": int(container)})    return {"image": image, "ports": ports}
Only image and ports are extracted from the compose file. All other machine metadata
(name, description, difficulty, hints) must be provided either in the web UI form or in a
companion koth-machine.yaml uploaded alongside the compose file.
4. Upload System
4.1 FastAPI File Upload
The upload API is built on FastAPI’s UploadFile class, which is a subclass of Starlette’s
UploadFile. It stores uploaded files as temporary SpooledTemporaryFile objects — files
under spool_max_size bytes stay in memory; larger files spill to disk automatically.
Reference: fastapi.tiangolo.com/tutorial/request-files
# platform-server/api/upload.py
from fastapi import APIRouter, UploadFile, File, Form, HTTPException
from typing import Annotated

    #    UploadFile.size is set by Starlette from Content-Length if provided.
    #    Do not rely on it alone — validate after reading.
    if yaml_file.size and yaml_file.size > 1024 * 1024:
        raise HTTPException(422, "YAML file must be under 1 MB")
    # 2. Read YAML — always use safe_load (see Section 12.1)
    raw = await yaml_file.read()
    try:
        spec = yaml.safe_load(raw)Why shutil.copyfileobj instead of await asset.read():
await asset.read() loads the entire file into memory at once. shutil.copyfileobj
streams in chunks (default 16 KB). For Docker image tarballs that can be several GB, always
use copyfileobj.

5.2 Remote Deployment via SSH + Docker
Rather than exposing Docker over TCP, the platform deploys to remote nodes by SSHing in
and running Docker commands over the SSH channel. This avoids needing to configure TLS
certificates on every node and keeps the Docker daemon bound to its local socket only.
The deployer uses Paramiko for SSH.
Install: pip install paramiko
Reference: paramiko.org/api/client.html
# platform-server/engine/deployer.py
import paramiko
import json
def deploy_to_node(
    node_host: str,
    ssh_user: str,
    ssh_key_path: str,    image: str,
    container_name: str,
    port_mapping: dict,  # {host_port: container_port}
    king_file: str,
) -> str:
    """
    Deploy a container on a remote node via SSH.
    Returns the Docker container ID on success.
    Raises RuntimeError on failure.
    """
    ssh = paramiko.SSHClient()
    # AutoAddPolicy accepts any host key.
    # For production, use RejectPolicy and maintain a known_hosts file.
    # See Section 12.2 for the security tradeoff.
    ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
    try:
        ssh.connect(
            hostname=node_host,
            username=ssh_user,
            key_filename=ssh_key_path,
            timeout=10,          # connection timeout in seconds
            banner_timeout=10,   # time to wait for SSH banner
        )
        # Build port flags: "-p 10085:80"
        port_flags = " ".join(
            f"-p {hp}:{cp}" for hp, cp in port_mapping.items()
        )
        # Pull the image first, separate from run
        pull_cmd = f"docker pull {image}"
        _exec(ssh, pull_cmd, timeout=120)  # image pulls can be slow
        # Run the container
        run_cmd = (
            f"docker run -d "
            f"--name {container_name} "
            f"--restart unless-stopped "
            f"{port_flags} "
            f"{image}"
        )
        stdout = _exec(ssh, run_cmd, timeout=30)
        container_id = stdout.strip()
        return container_id    finally:
        ssh.close()
def _exec(ssh: paramiko.SSHClient, command: str, timeout: int) -> str:
    """
    Execute a command over SSH and return stdout.
    Raises RuntimeError if the command exits non-zero.
    IMPORTANT: Read stderr before or concurrently with stdout to avoid
    the deadlock described in paramiko issue #1778.
    See Section 11.2 for full explanation.
    """
    stdin, stdout, stderr = ssh.exec_command(command, timeout=timeout)
    stdin.close()  # we never write to stdin
    # Read stderr first (or at least drain it) to prevent the
    # transport-layer deadlock when both buffers fill up.
    err = stderr.read().decode("utf-8", errors="replace")
    out = stdout.read().decode("utf-8", errors="replace")
    exit_code = stdout.channel.recv_exit_status()
    if exit_code != 0:
        raise RuntimeError(
            f"Command failed (exit {exit_code}).\n"
            f"Command: {command}\n"
            f"Stderr: {err}\n"
            f"Stdout: {out}"
        )
    return out

# platform-server/engine/builder.py
import docker, os
def build_and_push(
    dockerfile_dir: str,
    tag: str,
    registry_host: str,  # e.g. "localhost:5000" for a local registry
) -> str:    """
    Build a Docker image from a Dockerfile directory.
    Push to a local registry so nodes can pull it.
    Returns the full image reference: registry_host/tag
    """
    client = docker.from_env()
    full_tag = f"{registry_host}/{tag}"
    # client.images.build returns (Image, generator_of_log_lines)
    # The generator must be consumed to avoid the build hanging.
    image, logs = client.images.build(
        path=dockerfile_dir,
        tag=full_tag,
        rm=True,           # remove intermediate containers after build
        forcerm=True,      # remove even on failure
    )
    # Consume and surface build logs
    for chunk in logs:
        if "stream" in chunk:
            print(chunk["stream"], end="")
        if "error" in chunk:
            raise RuntimeError(f"Build failed: {chunk['error']}")
    # Push to local registry
    # Requires a local registry running at registry_host.
    # Start one with: docker run -d -p 5000:5000 registry:2
    push_output = client.images.push(full_tag)
    print(push_output)
    return full_tag

def undeploy_from_node(
    node_host: str,
    ssh_user: str,
    ssh_key_path: str,
    container_name: str,
):
    ssh = paramiko.SSHClient()
    ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
    ssh.connect(node_host, username=ssh_user, key_filename=ssh_key_path)
    try:
        # Stop then remove. Separate commands so stop errors don't
        # prevent removal of an already-exited container.
        try:
            _exec(ssh, f"docker stop {container_name}", timeout=30)
        except RuntimeError:
            pass  # container may already be stopped
        _exec(ssh, f"docker rm {container_name}", timeout=10)
    finally:
        ssh.close()
6. Port Manager
6.1 Problem
When a machine is deployed, the platform must assign it a host port on the target node.
Two concurrent deployments to the same node must not receive the same port. The
assignment must survive a platform server restart.
6.2 Design
The port manager uses SQLite’s UNIQUE constraint on (node_host, port) as an atomic
reservation mechanism. There is no in-memory lock — the database is the lock. This works
correctly even with multiple concurrent API requests because SQLite serializes writes.# platform-server/engine/port_manager.py
import sqlite3, datetime
PORT_RANGE_START = 10000
PORT_RANGE_END   = 20000
def reserve_port(db_path: str, node_host: str, preferred_port: int | None = None) -> int:
    """
    Reserve the next available port on node_host.
    If preferred_port is given and free, reserves that port.
    Returns the reserved port number.
    Raises RuntimeError if no ports are available.
    """
    with sqlite3.connect(db_path) as conn:
        conn.execute("PRAGMA journal_mode=WAL")
        conn.execute("PRAGMA busy_timeout=5000")
        # Fetch all currently reserved ports for this node
        taken = {
            row[0] for row in conn.execute(
                "SELECT port FROM port_reservations WHERE node_host = ?",
                (node_host,)
            )
        }
        # Also check the deployments table for active containers
        active = {
            row[0] for row in conn.execute(
                "SELECT host_port FROM deployments WHERE node_host = ? AND stopped_at IS NULL",
                (node_host,)
            )
        }
        reserved = taken | active
        # Try preferred port first
        candidates = []
        if preferred_port and preferred_port not in reserved:
            candidates.append(preferred_port)
        candidates += [p for p in range(PORT_RANGE_START, PORT_RANGE_END) if p not in reserved]
        if not candidates:
            raise RuntimeError(f"No free ports on {node_host}")
        chosen = candidates[0]
        conn.execute(
            "INSERT INTO port_reservations (node_host, port, reserved_at) VALUES (?, ?, ?)",
            (node_host, chosen, datetime.datetime.utcnow().isoformat())6.3 Reserved Ports

7. Adaptive Referee
7.1 What Changed from v1
In v1, the referee loaded SSH targets from NODE_SSH_TARGETS in .env at startup. Those
targets were static for the entire competition.
In v2, the referee calls GET /api/machines/active at the start of each scoring interval. This
returns the current list of deployed, active machines. The referee diffs this against its
current poll list and adds or removes machines without restarting.

# referee-server/poller.pyimport asyncio, paramiko, httpx, datetime, logging
logger = logging.getLogger(__name__)
class Poller:
    def __init__(self, platform_url: str, ssh_key_path: str, interval_seconds: int):
        self.platform_url = platform_url
        self.ssh_key_path = ssh_key_path
        self.interval = interval_seconds
        self._active_machines: dict[str, dict] = {}  # machine_id → machine info
    async def run_forever(self):
        while True:
            try:
                await self._reload_machines()
                await self._poll_all()
            except Exception as e:
                logger.error(f"Poller error: {e}", exc_info=True)
            await asyncio.sleep(self.interval)
    async def _reload_machines(self):
        """Fetch the current active machine list from the registry."""
        async with httpx.AsyncClient() as client:
            resp = await client.get(
                f"{self.platform_url}/api/machines/active",
                timeout=5.0
            )
            resp.raise_for_status()
            machines = {m["id"]: m for m in resp.json()}
        added   = set(machines) - set(self._active_machines)
        removed = set(self._active_machines) - set(machines)
        for mid in added:
            logger.info(f"Referee: tracking new machine {machines[mid]['name']} "
                        f"at {machines[mid]['node_host']}:{machines[mid]['host_port']}")
        for mid in removed:
            logger.info(f"Referee: dropping machine {self._active_machines[mid]['name']}")
        self._active_machines = machines
    async def _poll_all(self):
        """Read king.txt on every active machine and submit scores."""
        tasks = [self._poll_one(m) for m in self._active_machines.values()]
        results = await asyncio.gather(*tasks, return_exceptions=True)

                if owner:

    async def _poll_one(self, machine: dict) -> tuple[str | None, str]:
        """
        SSH to the machine's host node; read king_file.
        Returns (owner_team_name, timestamp) or (None, timestamp) if unowned.
        Runs in a thread pool because Paramiko is synchronous.
        """
        loop = asyncio.get_event_loop()
        return await loop.run_in_executor(None, self._ssh_read_king_file, machine)
    def _ssh_read_king_file(self, machine: dict) -> tuple[str | None, str]:
        """Synchronous SSH read. Called from executor thread."""
        ssh = paramiko.SSHClient()
        ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
        timestamp = datetime.datetime.utcnow().isoformat()
        try:
            ssh.connect(
                hostname=machine["node_host"],
                username=machine["node_ssh_user"],
                key_filename=self.ssh_key_path,
                timeout=5,
                banner_timeout=5,
            )
            # Use docker exec to read the file from inside the container.
            # This avoids needing SSH inside the container itself.
            cmd = (
                f"docker exec {machine['container_id']} "
                f"cat {machine['king_file']} 2>/dev/null || true"
            )
            _, stdout, stderr = ssh.exec_command(cmd, timeout=5)
            stderr.read()  # drain stderr to prevent deadlock (paramiko #1778)
            content = stdout.read().decode("utf-8", errors="replace").strip()
            owner = content if content else None
            return (owner, timestamp)
        except (paramiko.SSHException, OSError) as e:
            raise RuntimeError(f"SSH error for {machine['name']}: {e}") from e7.3 GET /api/machines/active Response Shape

8. Database Layer
8.1 SQLite + WAL Mode
The platform uses SQLite as its database. SQLite is appropriate here because:
The platform server is a single process running on one host
Write volume is low (one write per deployment, one write per scoring tick per machine)
No distributed state is needed
No separate database server to manage
WAL mode (Write-Ahead Logging) is enabled on every connection. Under the default
DELETE journal mode, any writer blocks all readers. WAL allows multiple concurrent readers
while a single writer appends to a separate WAL file.
Reference: sqlite.org/wal.html
# platform-server/engine/db.py
import sqlite3
def get_connection(db_path: str) -> sqlite3.Connection:
        finally:
            ssh.close()    conn = sqlite3.connect(db_path, check_same_thread=False)
    conn.row_factory = sqlite3.Row  # access columns by name
    # Enable WAL mode. This is persistent — only needs to be set once
    # per database file, but setting it on each new connection is harmless.
    conn.execute("PRAGMA journal_mode=WAL")
    # Without busy_timeout, a concurrent write raises
    # sqlite3.OperationalError: database is locked immediately.
    # 5000 ms is enough for the other writer to finish under normal load.
    conn.execute("PRAGMA busy_timeout=5000")
    return conn
WAL limitations to be aware of:
WAL requires that all processes accessing the database are on the same host machine.
It will not work correctly over NFS or other network filesystems. If you mount the
database file over the network, use DELETE journal mode instead.
WAL creates two additional files alongside the database: platform.db-wal and
platform.db-shm. These must be kept with the .db file. Deleting them while the
database is in use causes corruption.
WAL files grow until a checkpoint writes them back to the main database file. A passive
checkpoint happens automatically, but you can run PRAGMA wal_checkpoint(TRUNCATE)
during a maintenance window to reclaim disk space.

11. Known Issues and Pitfalls
This section documents real problems encountered in development and production, with
references to upstream issue trackers where applicable.
11.1 Docker SDK: Port Mapping Argument Order
The Docker SDK for Python uses an inverted port notation compared to the CLI. In the CLI
you write docker run -p HOST:CONTAINER. In the SDK, the dictionary key is the container
port and the value is the host port:
# CLI:  docker run -p 10085:80
# SDK:
client.containers.run(
    image,
    ports={"80/tcp": 10085}  # container_port/protocol: host_port
)
This is explicitly noted in the SDK documentation. Getting it backwards results in the
container binding to port 80 on the host (or failing if port 80 is taken) rather than port
10085.
Reference: docker-py.readthedocs.io/en/stable/containers.html
11.2 Paramiko: stdout.read() Deadlock
ssh.exec_command() returns (stdin, stdout, stderr) channel file objects. If you call
stdout.read() before reading stderr, and the remote command produces output on both
channels simultaneously, you can hit a transport-layer deadlock:
The remote process fills the stderr buffer (window size ≈ 2 MiB)
The SSH transport thread blocks trying to write more stderrYour application thread blocks reading stdout
Neither side makes progress
The fix: Always read stderr before or instead of letting it buffer up. The safest pattern is
to read stderr first, then stdout:
# WRONG — can deadlock if command writes to both channels
out = stdout.read()  # blocks if stderr buffer is full
err = stderr.read()
# CORRECT — drain stderr first
err = stderr.read()
out = stdout.read()
Alternatively, pass get_pty=True to exec_command to merge stdout and stderr into a single
channel (PTY), which eliminates the two-buffer deadlock but changes output formatting.
Reference: github.com/paramiko/paramiko/issues/1778
11.3 Paramiko: exec_command Does Not Return Until EOF
stdout.read() blocks until the remote command closes its stdout. Commands that hold
stdout open (interactive shells, daemons, tail -f) will block indefinitely. Always set a
timeout on exec_command:
stdin, stdout, stderr = ssh.exec_command(command, timeout=30)
The timeout is applied to channel read operations. If the command does not close stdout
within timeout seconds of inactivity, paramiko.buffered_pipe.PipeTimeout is raised.
This is different from the connection timeout set in ssh.connect(timeout=...). The
connection timeout applies to establishing the TCP connection. The channel timeout applies
to waiting for data on the channel.
11.4 SQLite: WAL Files Must Not Be Separated from the Database
If you copy the database file without copying platform.db-wal and platform.db-shm,
uncommitted transactions are lost and the database may be corrupt. Use the SQLite backup
API (via .backup()) or the VACUUM INTO command to create a consistent backup:
# Safe backup using the SQLite CLI
sqlite3 platform.db "VACUUM INTO '/backup/platform_backup.db'"Reference: sqlite.org/wal.html — “Avoiding Resource Starvation”
11.5 SQLite: WAL Does Not Work on Network Filesystems
WAL mode requires shared memory primitives (platform.db-shm) that do not work reliably
over NFS, CIFS, or other network filesystems. If the database file is on a network-mounted
volume, WAL mode will silently fall back to rollback journal mode or cause corruption. Keep
the SQLite database on a local filesystem.
Reference: sqlite.org/wal.html — “Limitations”
11.6 Docker Build Context Size
When building from a Dockerfile with client.images.build(path=...), the Docker SDK
sends the entire contents of path to the Docker daemon as a build context tarball. If the
directory contains large files (e.g., database dumps, VM images), this will be slow and may
exhaust memory on the platform server.
Always include a .dockerignore file at the root of the build context directory listing files to
exclude. The syntax is identical to .gitignore:
# .dockerignore
*.tar
*.iso
*.vmdk
node_modules/
__pycache__/
11.7 Container Names Must Be Unique Per Node
docker run --name fails if a container with that name already exists on the node, even if it
is stopped. The deployer generates container names as koth-{machine_id[:8]} and calls
docker rm on undeploy. If a deploy fails midway (e.g., the image pulled but the run
command failed), the partial container may be left behind with the name reserved. The
recovery flow must call docker rm -f {container_name} before retrying.
# deployer.py: before every deploy attempt, clean up any stale container

12. Security Considerations
12.1 PyYAML: Never Use yaml.load() on Untrusted Input
PyYAML’s yaml.load() function can execute arbitrary Python code if given a malicious
YAML file. This is not a bug — it is documented behaviour. The
!!python/object/apply:os.system YAML tag, among others, allows code execution. This
affects yaml.load() with the default FullLoader in versions before 5.4 and with
Loader=yaml.Loader in all versions.
Always use yaml.safe_load() for any organizer-uploaded content. safe_load() only
constructs standard Python objects (dicts, lists, strings, numbers, booleans, None) and
rejects any tag-based object construction.
import sqlite3
def reserve_port_with_retry(db_path, node_host, preferred_port=None, max_attempts=5):
    for attempt in range(max_attempts):
        try:
            return reserve_port(db_path, node_host, preferred_port)
        except sqlite3.IntegrityError:
            preferred_port = None  # clear preference; pick any free port
            continue
    raise RuntimeError(f"Could not reserve a port after {max_attempts} attempts")
# NEVER do this with organizer-uploaded files
config = yaml.load(user_data)                              # dangerous
config = yaml.load(user_data, Loader=yaml.Loader)          # dangerous
config = yaml.load(user_data, Loader=yaml.FullLoader)      # dangerous (pre-5.4)
# Always do this
config = yaml.safe_load(user_data)                         # safe in all versionsCVE references: CVE-2017-18342, CVE-2020-1747, CVE-2020-14343
Reference: github.com/yaml/pyyaml/wiki/PyYAML-yaml.load(input)-Deprecation
12.2 Paramiko: Host Key Verification
Using AutoAddPolicy (accept any host key on first connection) is vulnerable to a man-in-
the-middle attack: an attacker on the same network can intercept the SSH connection on
the first deploy and present a forged key. The platform will accept it, and subsequent
connections to the same host will trust that forged key.
For a competition network where you control all nodes, the acceptable mitigation is to pre-
populate ~/.ssh/known_hosts on the platform server with each node’s host key before
running the platform, and use RejectPolicy instead:
ssh.set_missing_host_key_policy(paramiko.RejectPolicy())
# Pre-populate known_hosts before running the platform:
# ssh-keyscan -H 192.168.1.10 >> ~/.ssh/known_hosts
# ssh-keyscan -H 192.168.1.11 >> ~/.ssh/known_hosts
# ssh-keyscan -H 192.168.1.12 >> ~/.ssh/known_hosts
12.3 Container Isolation
The machines on the platform are intentionally vulnerable. Participants will exploit them. Do
not run the platform on nodes that also run production services. Each node should be a
dedicated VM used exclusively for the competition.
Network-level isolation recommendations:
Participants can reach node_host:host_port for each deployed machine
Participants must not be able to reach port 22 on any node (this is how the referee
manages machines)
Participants must not be able to reach the platform server admin dashboard on port
8000
The firewall on each node should block all inter-container traffic unless the machine
requires it
12.4 Admin API Key
The ADMIN_API_KEY must be set in the platform server’s .env before starting. All admin
API endpoints check for this key in the X-Admin-Key header. The platform refuses to start
without it unless ALLOW_UNSAFE_NO_ADMIN_API_KEY=true is set (for local development only).Generate a key with sufficient entropy:
python3 -c "import secrets; print(secrets.token_hex(32))"
12.5 Docker Image Provenance
When an organizer provides image: someuser/somerepo:latest in their YAML, the deployer
will pull that image from Docker Hub onto your competition nodes. Malicious images could
exfiltrate network data, attack other containers, or harm the host. For public competitions:
Require organizers to submit images in advance for review
Pull and scan images on the platform server before allowing deploy
Use a content-addressable reference (image: myorg/vuln-nginx@sha256:a3f2...)
instead of a mutable tag to prevent the image from being replaced after review

14.2 Requirements for Contributed Machines
1. The machine must be self-contained. Everything needed to build and run it must be in
the directory. No references to external servers or private registries.
2. /root/king.txt must exist and be world-writable after privilege escalation. The file
is created at container startup with content unclaimed. A participant who gets root can
overwrite it.
3. The machine must have a two-stage exploit path: initial foothold → privilege
escalation to root. Single-step root is only acceptable for easy difficulty.
4. The README.md must include: initial vector, privesc vector, CVE references if
applicable, and a note on which services run on which container ports.
5. No outbound internet calls during runtime. The container must not depend on
external package downloads or API calls once running.
